## Methodology

1. `information_schema.tables` → all 48 `hr_%` tables in `public` schema.
2. Dynamic `SELECT count(*)` per table via Supabase TX pooler (6543, read-only).
3. Literal `grep -rlF <table>` across `frontend/src/` for UI references.
4. Paginated `GET /workflows?active=true` (nodes inline), regex `hr_[a-z_]+` filtered to the 48 known names.

## Honest limitations

- **UI grep is literal.** Tables accessed only via RPC, DB views, or dynamic string-built queries are undercounted. E.g. `hr_leave_types` shows 0 UI refs but is almost certainly surfaced indirectly through the leave UI's joins — treat its 'hidden' status as 'no dedicated admin screen', not 'completely invisible'.
- **Policy editing (V6) uses `hr_documents`, not `hr_policies`.** The verified V6 edit flow writes versioned rows to `hr_documents` (parent_document_id / is_current_version). `hr_policies` (0 rows) is a separate, lightly-used table referenced only by the policy-sync workflows.
- **Workflow regex counts references, not necessarily writes.** A reference may be a read, a write, or a commented node. Counts indicate *involvement*, not guaranteed active writes.
- Row counts are tenant-agnostic totals (all tenants), captured 2026-05-29.
