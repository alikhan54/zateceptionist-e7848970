## Method

adeel admin (zate tenant). Each test pairs UI action with a Supabase
REST query that proves the row actually changed. Test data was created
under controlled identifiers and cleaned up at end.

## Critical (blocking-merge) failures

None.

## Non-critical (can ship, fix later)

- **V6 SKIP**: Documents page didn't surface row dropdowns within the
  spec's 3.5s budget. DB has 1 policy doc for zate (`employment_contract`,
  `passport_copy`, etc. plus 1 `policy`). The page renders, just slowly.
  Not a feature break — the spec's locator needs to wait for lazy rows.
- **V10 UI flake**: AI Generate button click didn't render questions in
  the dialog within 35s. The underlying workflow demonstrably works
  (R3 verified end-to-end with `target_job_id` → 7 Claude questions
  PATCHed). The UI plumbing between button click and rendered Q-cards
  needs a longer test or a network-response wait.

## Unverifiable in this environment

- **Tenant isolation from inside Cosmique session**: I only had Zate
  admin credentials. The reverse direction (login as Cosmique, verify
  zate's "Senior React Engineer" jobs don't appear) was not exercised.
  By symmetry of the RLS / `tenant_id` filter, it should hold, but a
  full audit needs Cosmique creds.
