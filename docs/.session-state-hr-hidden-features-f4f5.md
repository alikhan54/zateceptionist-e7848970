Design specifics:
- Components are tenant-scoped on `useTenant().tenantConfig.id` (UUID-as-text),
  matching how `hr_employees` stores `tenant_id`.
- `NotificationsFeed` prop `scope`: `"mine"` filters `recipient_id = auth user id`
  (staff /my Alerts tab); `"all"` is tenant-wide (admin /hr/notifications).
- `recipient_id` has NO FK and the table is empty — chosen to be `auth.users.id`
  so the UI filter and the activation policy (`recipient_id = auth.uid()`) stay
  in lockstep.
- Shift CRUD (`ShiftDefinitionsTab`, `ShiftAssignmentsTab`) gates Add/edit/delete
  behind `canManage = isAdmin || isManager`. `days_of_week` uses ISO 1=Mon..7=Sun
  (matches the `{1,2,3,4,5}` column default on `hr_employee_shifts`).

### ⚠ hr_employee_shifts cross-tenant leak (must fix at activation)

`hr_employee_shifts` carries a single permissive policy `"Allow authenticated"`
with `USING (true) / WITH CHECK (true)` for ALL commands. That means **any
authenticated user of any tenant can read/write every row** in that table at the
database level. It has not surfaced because (a) the table is empty (0 rows) and
(b) the new `ShiftAssignmentsTab` filters every query by `tenant_id`. The
activation SQL **drops** this policy and replaces it with tenant isolation. Until
the activation SQL runs, do not seed cross-tenant data into this table.

## Activation SQL (review then apply post-merge — DO NOT EXECUTE from a session)

File: [`frontend/docs/.hr-rls-activation.sql`](.hr-rls-activation.sql)

Every policy in it is an EXACT mirror of the verified live `hr_employees`
`"Tenant isolation"` policy:

```
FOR ALL  USING ((auth.role() = 'service_role'::text)
                OR (tenant_id = (get_user_tenant_uuid())::text))
-- WITH CHECK intentionally omitted (Postgres reuses USING as the write-check)
```

- §1 `hr_notifications`: tenant-isolation (admin scope="all" + mark-read UPDATEs +
  service writes) **plus** a per-user SELECT policy `recipient_id = auth.uid()`
  (requested for the staff feed; additive — documented OR-semantics; a stricter
  staff-only variant is included commented-out).
- §2 `hr_shifts`: tenant-isolation only.
- §3 `hr_employee_shifts`: **DROP** `"Allow authenticated"` first, then
  tenant-isolation.
- §0 preflight + §4 verification queries included. Rollback block at the bottom
  intentionally does NOT recreate the leak.
