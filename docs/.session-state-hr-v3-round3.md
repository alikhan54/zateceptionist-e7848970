### R3.1 — Auto-fire Q-Gen on Job Create

- **Approach:** instead of modifying the 243-node `420 HR Part 2 v1.9.3`
  workflow (risky — too many edges), built a small standalone cron that
  catches recently-created jobs without questions and fires the gen
  webhook. Same outcome, zero risk to existing workflow.
- **New workflow:** `420 HR Job Q-Gen Auto-Fill v1.0` (`hDmRNaIrs1qJHuYq`),
  active. Schedule: every 5 minutes.
- **Logic:**
  1. Query `hr_job_requisitions` for rows created in last 10 min.
  2. Match rows where `interview_question_method IN ('auto', 'ai_generated')`
     OR (`interview_questions` is null/empty AND method != 'manual').
  3. For each, fire `/hr/job/generate-interview-questions` with
     `target_job_id`. The Q-Gen workflow PATCHes the row with the result.
- **Verified end-to-end:**
  - Inserted test job (Cosmique tenant, `interview_question_method='auto'`).
  - Hit the gen webhook with `target_job_id` directly.
  - 7 questions appeared on the row, `interview_question_method` flipped
    to `ai_generated`, first question correctly healthcare-flavored
    ("multiple patients checking in simultaneously…").
  - Test row deleted.

### R3.2 — DB Trigger Guard for Staff Profile Updates

- **Function:** `hr_employees_self_update_guard()` (SECURITY DEFINER).
- **Trigger:** `hr_employees_self_update_guard_trg`, BEFORE UPDATE.
- **Logic:**
  1. If `auth.uid()` is NULL (service-role / cron), pass through. ✅ verified.
  2. If caller is editing a row whose `OLD.user_id` ≠ caller, pass through
     (admin editing someone else).
  3. If caller's role ∈ {`master_admin`, `admin`, `manager`}, pass through.
  4. Otherwise: block changes to `first_name`, `last_name`, `company_email`,
     `work_email`, `position`, `job_title`, `department[_id]`, `salary`,
     `date_of_joining`, `employment_status`, `manager_id`, `tenant_id`,
     `user_id`, `employee_number`. Raise descriptive errors.
- **Security note:** check uses `OLD.user_id` (immutable from client's
  perspective) — staff can't bypass by setting `NEW.user_id` to a fake value
  (that would also be blocked by the user_id check itself).
- **Verified:** service-role can still update salary (auth.uid()=NULL path).
  Trigger present in `pg_trigger` table with tgenabled='O'.
