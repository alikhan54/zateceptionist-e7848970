## What was broken → root cause → fix

### 1A — Employee "Save Changes" failed (PRIORITY)
- **Root cause (two compounding):** (a) the dialog sent `department_name`, which is **not a column** on `hr_employees` (real: `department` + `department_id`) → `column "department_name" does not exist`; (b) save routed through the n8n webhook `/hr/employee/update` (workflow `U9Yb2JSa46NlOKXC`, no executions / Bug #96 backlog). Generic "Failed to update employee" hid the cause.
- **Fix:** `useHR.updateEmployee`/`terminateEmployee` → **direct supabase update**, whitelist of real columns only, `.eq('id').eq('tenant_id')`, `.maybeSingle()` guard, best-effort `hr_audit_logs` row, and the real `error.message` surfaced in the toast. `Employees.tsx` now resolves `department_name → {department, department_id}` before saving.

### 1B — Courses: JSON leak + no edit/delete/manual content
- **Root cause:** catalog card rendered `{program.provider}` as a vendor label. For AI courses `provider` is the JSON content blob → raw `{"ai_generated":true,"agent":"claude-premium",…}` shown. (Confirmed on cosmique's 2 AI courses: "Patient Confidentiality", "LIKE HANDLING THE PATIENTS COMPLAINTS". zate has 0, so it looked fine there.) No edit/delete/manual-content paths existed.
- **Fix:** `providerLabel()` renders a vendor **only** when it's plain text (never `{`/`[`); AI/Video badges instead. New `useTraining` mutations `updateCourse`, `deleteCourse` (soft-delete `status='cancelled'`, preserves enrollments), `addCourseMedia` (merges video/doc URL into the `provider` JSON, preserves vendor). Per-card actions menu (Edit / Add video-document / Delete-with-confirmation), Edit/Add-Media/Delete dialogs, soft-deleted courses filtered, description hardened against JSON, and a real "Add video / document" CTA in the empty course player.

### 1C — Reviews: blank recipient names + not editable
- **Root cause:** reviews query was a bare `select('*')` with **no name join** — `employee_name` was always undefined → blank. Also `createReview` inserted `status='draft'`, which **violates the DB CHECK** (`pending|in_progress|submitted|acknowledged`) → manual Create Review would fail.
- **Fix:** enrich each review with `employee_name` (id→name map from `hr_employees`) + `reviewed_on`; shared `renderReviewRow` shows **who** the review is for + type label + date + rating + status + Edit affordance; My Reviews wired to the signed-in user's own reviews (via `useCurrentEmployee`); Team Reviews shows real names; `getStatusBadge` maps the real 4 statuses; new `updateReview` mutation; `createReview` status `draft → in_progress`.
