## Part D — RLS security findings

### The leak mechanism (why this matters)

Postgres **permissive** RLS policies combine with **OR**. A table can carry a correct `tenant_id = get_user_tenant_id()` policy and *still* leak across tenants if it *also* carries a broad-role (`authenticated` / `public`) policy whose qualifier is literally `true`. The `true` policy wins for every row. This is the exact F5b / `hr_employee_shifts` hole — and it is **systemic**, not isolated.

Verified examples (USING/CHECK pulled live):

- `accounting_clients` — has correct `rls_tenant_read/update/delete` policies, **but** `rls_service_all` is `USING (true)` granted to PUBLIC. Allow-all wins.
- `clinic_consultations` — policy literally **named** `clinic_consultations_tenant_isolation` but its body is `USING (true) / CHECK (true)`. Misnamed; provides zero isolation.
- `hr_goals` — correct `Tenant isolation` policy defeated by a co-existing `Authenticated access` policy with `USING (true)`.

Policies scoped **only** to `service_role` are *excluded* from all leak counts below — `service_role` bypasses RLS anyway, so `USING(true)` there is the normal explicit-access pattern (273 such policies excluded).

## Methodology & limits

- **Leak classification is role-aware**: only `public`/`authenticated`/`anon` true-quals count; `service_role`-only true-quals are excluded as normal.
- **UI detection** = word-boundary match of the table name in `frontend/src/**`. A table named generically (e.g. `messages`, `calls`) can match unrelated code → UI count may be *over*-stated for short/common names (false 'has UI'). Conversely a table accessed only via an RPC/view alias would be *under*-counted.
- **Row counts** are point-in-time; an empty table today may be an active feature awaiting data.
- **Workflow detection** = table name appearing in an active workflow's node JSON; 176/268 active workflows scanned (92 fetch-failed).
- This audit changed **nothing** in the database. All findings are observations.
